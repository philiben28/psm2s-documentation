# L3.4.1 — Cartographie Core / Paramètre / Variante

Date : 04/07/2026
Statut : **analyse fonctionnelle, aucune décision technique**. Conformément
à la méthode fixée par le Directeur Technique pour L3.4, ce document ne
contient ni structure de dossiers, ni mécanisme Git, ni paramètre de
configuration technique — uniquement la cartographie fonctionnelle, à
valider avant d'aborder L3.4.2.

## Méthode

Cartographie construite à partir de l'examen du code réel de PSM2S
(`registre/models.py`, `config/settings.py`, `Templates/base.html`), pas
d'un modèle théorique. Pour chaque élément : commun à tous les clients
(Core), configurable sans code (Paramètre), ou spécifique à un client
(Variante) — en appliquant le principe directeur fixé par le DT :

> Tout ce qui peut être résolu par un paramètre ne doit jamais devenir une
> variante de code.

---

## 1. Cœur métier réglementaire — très majoritairement Core

Le registre de sécurité numérique couvre des obligations légales
applicables à **tout établissement recevant du public en France**, quel
que soit le client. Ce périmètre est générique par nature, pas propre à
l'ADPEP28 :

| Élément | Core | Paramètre | Variante |
| --- | :-: | :-: | :-: |
| Rôles utilisateurs (admin, responsable sécurité, directeur, factotum, prestataire) et permissions par établissement | ✅ | | |
| Établissements / Bâtiments, classification ERP (types J/R/U/W, catégories 1 à 5) | ✅ | | |
| Registre de contrôles réglementaires (`ControleEtablissement`), historique annuel des réalisations | ✅ | | |
| Catalogue des types de contrôles et périodicités (`TypeControle`, `Periodicite`) | | ✅ | |
| Tickets travaux et interventions (workflow À traiter → Clôturé) | ✅ | | |
| Contrats prestataires | ✅ | | |
| Commissions de sécurité (visites, prescriptions, suivi) | ✅ | | |
| Carnet sanitaire Eau (réseaux, points de prélèvement, analyses, seuils réglementaires) | ✅ | | |
| DUERP (unités de travail, sources de danger, évaluations, plan d'action) | ✅ | | |
| Registre Public d'Accessibilité (zones, 4 types de handicap, Ad'AP) | ✅ | | |
| Droits de signature par utilisateur/document/établissement | ✅ | | |
| Calendrier réglementaire (Mois/Semaine/Jour) | ✅ | | |

**Constat** : le catalogue des contrôles (`TypeControle`) et les
périodicités (`Periodicite`) sont **déjà** des données en base, pas du
code — ils sont donc déjà correctement traités comme Paramètre depuis
l'origine du projet, avant même que la question ne se pose formellement.
Les catégories de risque DUERP (`CategorieRisque`, `SourceDanger`) et les
zones d'accessibilité (`EvaluationZone.ZONE_CHOICES`) sont en revanche
codées en dur (choix Django) : elles restent Core au sens où leur contenu
est fixé par la réglementation nationale, identique pour tout client —
mais un futur client avec des besoins de personnalisation de ces listes
ferait basculer cet élément en Paramètre (non nécessaire aujourd'hui, à
surveiller).

## 2. Identité et branding — aujourd'hui du Core qui devrait être Paramètre

C'est ici que se trouve l'écart principal entre l'état actuel du code et
un vrai socle multi-clients. PSM2S n'a qu'un seul client aujourd'hui
(ADPEP28), et plusieurs éléments propres à ce client sont **codés en
dur**, alors qu'ils devraient être des paramètres par instance :

| Élément | État actuel dans le code | Cible |
| --- | --- | --- |
| Nom de l'organisation (`NOM_ORGANISATION = 'Patrimoine ADPEP28'`, `config/settings.py`) | Constante Python en dur | Paramètre |
| Domaine / `ALLOWED_HOSTS`, `BASE_URL` | Constantes Python en dur | Paramètre (déjà par variable d'environnement pour la partie infra — voir PROC-001/DEPLOIEMENT_o2switch.md — mais le nom de domaine reste en dur dans `settings.py`) |
| Identité d'envoi des emails (`DEFAULT_FROM_EMAIL`, `EMAIL_HOST_USER`) | Constantes Python en dur | Paramètre |
| Logo, nom affiché « PSM2S », couleurs de la sidebar/thème (`Templates/base.html`, variables CSS `--accent`, `--side-bg`…) | Codées en dur dans le template | Paramètre |
| Coordonnées de contact affichées (si utilisées dans les documents générés, PDF, emails) | À vérifier au cas par cas, non cartographié en détail ici | Paramètre |

**Aucune de ces constatations ne demande une action immédiate** — c'est
un inventaire, pas une liste de tâches. Mais c'est la preuve concrète que
« Core » aujourd'hui contient déjà, sans que ce soit voulu, des éléments
propres à un seul client. Une vraie architecture multi-clients devra
sortir ces éléments du code vers un mécanisme de paramétrage (nature du
mécanisme = décision technique, hors périmètre de L3.4.1).

## 3. Fonctionnalités activables — gap identifié, aucun mécanisme aujourd'hui

Aujourd'hui, **tous les modules sont toujours actifs pour tous les
rôles** : le menu de navigation (`Templates/base.html`) n'a aucune notion
de « module désactivé pour ce client ». Si un futur client n'avait pas
besoin, par exemple, du Carnet Eau ou des Commissions de sécurité, rien
dans le code actuel ne permettrait de masquer ce module sans le retirer
en dur.

Ce n'est pas un défaut du Lot 3 — c'est une conséquence naturelle du fait
que PSM2S n'a eu qu'un seul client jusqu'ici. C'est en revanche **le
premier vrai sujet de conception pour L3.4.2/L3.4.3** : décider si les
modules ci-dessous doivent devenir des fonctionnalités activables par
paramètre plutôt que des blocs toujours présents :

- Carnet sanitaire Eau
- Commissions de sécurité
- DUERP
- Registre d'Accessibilité
- Tickets travaux / Interventions

Aucune décision n'est prise ici : ce constat est reporté tel quel pour
L3.4.2 (niveaux de personnalisation) et L3.4.3 (règles d'évolution).

## 4. Variantes de code — aucune identifiée aujourd'hui

Examen fait, **aucun élément du code actuel ne constitue une variante
propre à un client** au sens strict (code différent selon le client). Ce
constat confirme ce qui était pressenti en amont de L3.4 : PSM2S est
aujourd'hui une application mono-client, pas une famille de variantes.
Le travail de L3.4 consiste donc à **anticiper** la personnalisation
métier (workflows particuliers, champs supplémentaires, rapports
spécifiques) plutôt qu'à corriger une dérive déjà existante — ce qui est
la situation la plus favorable pour concevoir un mécanisme propre dès le
départ, sans dette à résorber.

## Synthèse

- **Cœur métier réglementaire** (contrôles, DUERP, accessibilité, eau,
  commissions, tickets, rôles/permissions) : **Core**, légitimement
  commun à tout client ERP français. Estimation cohérente avec l'objectif
  du DT (90-95 % du logiciel resterait commun).
- **Catalogue de contrôles et périodicités** : déjà traité comme
  Paramètre (données, pas code) — bonne pratique préexistante à
  confirmer/généraliser.
- **Identité et branding** (nom, domaine, email, logo, couleurs) :
  aujourd'hui Core en dur, **candidat naturel à devenir Paramètre**.
- **Activation de modules** : aucun mécanisme aujourd'hui — gap identifié,
  à traiter en L3.4.2/L3.4.3, pas de décision prise ici.
- **Variantes de code** : aucune aujourd'hui — situation favorable, rien
  à corriger, seulement à anticiper.

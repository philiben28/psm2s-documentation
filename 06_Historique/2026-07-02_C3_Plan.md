# Plan / architecture — Lot C3 : Protection des médias et desserte sécurisée

Destinataire : Directeur Technique
Rédacteur : Lead Developer PSM2S
Date : 02/07/2026
Nature : **plan uniquement** (aucun code). Lot le plus structurant de la Phase 2. Débloque C1.

---

## 1. Problème (rappel)

Les fichiers `/media/**` (PV de commission, contrats, rapports, analyses, **signatures manuscrites**, pièces RGPD) sont aujourd'hui servis par `static(MEDIA_URL, …)` dans `config/urls.py`, **actif tant que `DEBUG=True`**, donc **accessibles sans authentification** à qui connaît/devine l'URL (anomalie C3, Critique). Les noms de fichiers sont prévisibles.

De plus, `static()` ne sert plus rien quand `DEBUG=False` → C1 ne peut pas basculer sans que C3 ait refait la desserte. C1 et C3 sont couplés.

## 2. Cible (conforme Architecture v1 §4.3)

- **Aucun média servi sans authentification.**
- Desserte via une **vue Django** qui **vérifie le périmètre** de l'utilisateur (réutilise `verifier_acces_etablissement` / `_get_etab_ids_autorises` — l'acquis de la campagne C4).
- Accès direct Apache à `/media` **bloqué** (seul Django sert les fichiers).
- Étape 1 : **streaming Python** (`FileResponse`). Étape 2 (ultérieure, décision DT déjà prise) : optimisation `X-Sendfile` si o2switch le permet.

## 3. Conception proposée

### 3.1 Interception centralisée (sans toucher les templates)

`MEDIA_URL` reste `/media/`. Les templates continuent d'utiliser `{{ obj.fichier.url }}` → `/media/…` **inchangés**. On remplace, dans `config/urls.py`, la desserte `static(MEDIA_URL,…)` par une route vers une **vue protégée** :
```
path('media/<path:chemin>', views.media_protegee, name='media_protegee')
```
Avantage : **zéro changement de template**, un seul point d'entrée, fonctionne identiquement en dev et en prod (la vue n'est pas conditionnée par `DEBUG`).

### 3.2 Résolution chemin → établissement (routage par préfixe)

Chaque modèle porteur de fichier a un préfixe `upload_to` **distinct**, ce qui permet un routage déterministe :

| Préfixe du chemin | Modèle / champ | Établissement (périmètre) |
|---|---|---|
| `documents/` | `Document.fichier` | `controle.etablissement_id` |
| `realisations/` | `RealisationControle.document` | `controle.etablissement_id` |
| `tickets/` | `PieceJointeTicket.fichier` | `ticket.etablissement_id` |
| `contrats/` | `Contrat.document` | `etablissement_id` |
| `commissions/` | `VisiteCommission.document_pv` | `etablissement_id` |
| `prescriptions/` | `ActionPrescription.document` | `prescription.visite.etablissement_id` |
| `analyses_eau/` | `AnalyseEau.document` | `point.reseau.etablissement_id` |
| `accessibilite/pieces/` | `PieceAdministrative.fichier` | `registre.etablissement_id` |
| `signatures/` (+ `signatures/duerp/`, `signatures/accessibilite/`) | signatures | **pas d'établissement** — cas particulier (voir §3.3) |

La vue : identifie le modèle par préfixe → retrouve l'instance par son chemin (`Model.objects.filter(<champ>=chemin).first()`) → si trouvée, `verifier_acces_etablissement(user, <etab_id>)` → sert le fichier ; sinon `Http404`. Un petit **registre** `(préfixe → modèle, champ, fonction d'accès)` rend l'ensemble lisible et extensible.

### 3.3 Cas des signatures (fichiers sans établissement)

Les images de signature (`signatures/…`) sont sensibles mais ne se rattachent pas directement à un établissement. **Décision à prendre** (§5) :
- **Option S1** : accès réservé aux **utilisateurs authentifiés** (ferme la fuite publique — objectif C3), sans filtrage fin.
- **Option S2** : plus strict — la signature personnelle n'est servie qu'à son propriétaire ; les signatures de documents (DUERP/accessibilité) suivent le périmètre de l'établissement du document. Plus sûr, un peu plus de code.

### 3.4 Desserte et en-têtes

- `FileResponse` en streaming depuis `MEDIA_ROOT`.
- `Content-Disposition: attachment` (téléchargement forcé) → empêche l'exécution inline de HTML/SVG piégés (bénéfice de sécurité, recoupe l'anomalie E3 sur les uploads).
- Type MIME déterminé prudemment.
- 404 (et non 403) hors périmètre, cohérent avec toute la campagne C4.

### 3.5 Blocage de l'accès direct Apache (action serveur)

Pour que Django soit **le seul** à servir `/media`, l'accès direct doit être bloqué au niveau Apache. Cible : soit **déplacer `MEDIA_ROOT` hors du webroot**, soit une règle **`.htaccess`** interdisant l'accès direct à `/media`. À coordonner avec l'exploitant (comme la rotation C2). Je fournirai la règle ; l'application côté serveur reste de ta responsabilité.

## 4. Impact

**Fichiers modifiés (prévision)**
- `config/urls.py` : remplacement de `static(MEDIA_URL,…)` par la route `media_protegee`. (Statique inchangé dans ce lot.)
- `registre/views.py` : nouvelle vue `media_protegee` + petit registre de routage (réutilise `verifier_acces_etablissement`).
- `registre/tests.py` : tests d'accès média (autorisé / hors périmètre / non authentifié / signatures).
- Documentation (`DEPLOIEMENT_o2switch.md`) : règle Apache de blocage `/media`, note X-Sendfile (Étape 2).
- **Aucun modèle, aucune migration, aucun template.**

**Risque de régression**
- Faible côté code (une vue additive + une route). Les liens `.url` continuent de fonctionner.
- Point de vigilance : bien couvrir **tous** les préfixes (sinon un fichier non routé renverrait 404 pour tout le monde). Les tests couvriront chaque préfixe.
- Coordination serveur (blocage Apache) nécessaire au déploiement, comme pour C2.

## 5. Décisions requises du DT

1. **Interception** : route centralisée sur `/media/<path>` (zéro template touché) — **recommandé** — ou vues de téléchargement par objet (plus explicites mais nombreuses + templates modifiés) ?
2. **Signatures** : Option **S1** (authentifié suffit) ou **S2** (propriétaire / périmètre du document) ? (recommandation : S2 si tu veux le niveau le plus strict ; S1 si on veut d'abord fermer la fuite et affiner ensuite.)
3. **Blocage Apache** : déplacer `MEDIA_ROOT` hors webroot **ou** règle `.htaccess` deny sur `/media` ? (recommandation : `.htaccess` deny, moins invasif ; déplacement hors webroot en cible idéale si o2switch le permet aisément.)
4. **Découpage** : traiter C3 en **un seul lot**, ou le scinder (C3-1 : infrastructure + fichiers rattachés à un établissement ; C3-2 : signatures) pour rester sur des incréments courts ? (recommandation : dépend du choix S1/S2 — si S2, un découpage peut aider.)

## 6. Position dans le séquencement

C3 est le dernier lot structurant. Une fois livré et déployé (avec blocage Apache), la desserte des fichiers ne dépend plus de `DEBUG` → **C1 devient une simple mise en service** (`DEBUG=False` + `check --deploy`), suivie de **E4** (cookies/HSTS).

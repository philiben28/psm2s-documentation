# Plan / architecture — Lot C3-2 : Politique d'autorisation des signatures (S2)

Destinataire : Directeur Technique
Rédacteur : Lead Developer PSM2S
Date : 03/07/2026
Nature : **plan uniquement** (aucun code). Termine le chantier média (fait suite à C3-1).

---

## 1. Objectif

Remplacer la desserte **transitoire** des signatures (C3-1 : authentification seule) par leur **politique définitive S2** :

| Signature | Fichier (`upload_to`) | Politique cible |
|---|---|---|
| Personnelle (`Utilisateur.signature`) | `signatures/` | **Propriétaire uniquement** |
| DUERP (`DUERP.signature_directeur/_responsable`) | `signatures/duerp/` | **Périmètre du DUERP** (établissement) |
| Accessibilité (`RegistreAccessibilite.signature_directeur/_responsable`) | `signatures/accessibilite/` | **Périmètre du registre** (établissement) |

Après C3-2, plus aucune exception dans le modèle de sécurité média.

## 2. Constat de stockage (vérifié)

- Les signatures de documents sont des **copies** créées à la signature (`_copier_signature`) et stockées sous `signatures/duerp/…` et `signatures/accessibilite/…`. Elles se rattachent donc à un **établissement** (celui du DUERP / du registre) → **exactement le modèle de périmètre déjà en place** (comme les documents métier de C3-1).
- La signature **personnelle** (`signatures/<fichier>`) n'est affichée que sur le **profil de son propriétaire** (`mon_profil`). Les documents signés affichent la **copie** (`signatures/duerp|accessibilite/…`), pas l'originale. → une règle **« propriétaire »** ne casse aucun autre affichage.

## 3. Conception proposée

### 3.1 Deux des trois cas rentrent déjà dans le modèle C3-1

`signatures/duerp/` et `signatures/accessibilite/` sont **basées sur l'établissement** : ce sont de nouvelles entrées `perimetre=True` avec un résolveur, **strictement comme les 8 familles de documents**. Résolveurs :
- `signatures/duerp/` → `DUERP` dont `signature_directeur` **ou** `signature_responsable` == chemin → `etablissement_id`.
- `signatures/accessibilite/` → `RegistreAccessibilite` (idem) → `etablissement_id`.

### 3.2 Le seul cas nouveau : la signature personnelle (« propriétaire »)

`signatures/<fichier>` (personnelle) n'est **pas** liée à un établissement mais à un **propriétaire**. Le modèle actuel du registre (`résolveur → etablissement_id → verifier_acces_etablissement`) ne couvre pas ce cas.

**Proposition (recommandée) — unifier l'autorisation dans le registre.**
Faire évoluer `RegleMedia` pour porter une fonction d'autorisation explicite :
`autoriser(user, chemin) -> bool`. `media_access` fournit deux fabriques :
- `par_perimetre(resolveur)` → autorise si `acces_etablissement_autorise(user, resolveur(chemin))` ;
- `par_proprietaire()` → autorise si l'utilisateur est le propriétaire de la signature (`Utilisateur.objects.filter(signature=chemin, pk=user.pk).exists()`).

La vue `media_protegee` devient encore **plus mince** : elle appelle simplement
`if not regle.autoriser(request.user, chemin): raise Http404`. Elle reste
totalement **ignorante des modèles** — toute la logique vit dans `media_access`.

Ordre du registre (important) : les préfixes **spécifiques** d'abord
(`signatures/duerp/`, `signatures/accessibilite/`) **avant** `signatures/`
(personnelle), car `regle_pour_chemin` retourne le premier préfixe correspondant.

### 3.3 Disposition

Toutes les signatures restent servies **`inline`** (affichées en `<img>`). Inchangé.

### 3.4 Fin du régime transitoire

Après C3-2, il n'y a plus de desserte « authentifié seul » pour les signatures.
Un préfixe inconnu reste en **404** (aucun fichier légitime n'est hors registre).

## 4. Impact

**Fichiers (prévision)**
- `registre/media_access.py` : évolution de `RegleMedia` (autorisation unifiée) + 2 résolveurs signatures + 1 vérificateur propriétaire + réordonnancement des entrées `signatures*`.
- `registre/views.py` : `media_protegee` — remplacement de la branche `perimetre` par un unique appel `regle.autoriser(...)` (la vue **se simplifie**, reste model-agnostic).
- `registre/tests.py` : tests des 3 cas (propriétaire OK / autre utilisateur 404 ; DUERP périmètre ; accessibilité périmètre).
- Documentation (`DEPLOIEMENT_o2switch.md`, CHANGELOG) : signatures passées au niveau définitif.
- **Aucun modèle, migration, template ni URL de template.**

**Risque de régression** : faible. Les documents métier (C3-1) conservent exactement le même comportement (ils passent par `par_perimetre`, équivalent à l'actuel). Seules les signatures changent de politique. Non-régression couverte par les 104 tests + nouveaux tests.

## 5. Décisions requises du DT

1. **Unification de l'autorisation** dans le registre (`autoriser(user, chemin)`), au prix d'une **simplification** de la vue (une seule délégation) — **recommandé** — ou conserver la structure C3-1 (`perimetre`+`resolveur`) et ajouter une branche spécifique « propriétaire » dans la vue (deux mécanismes, moins unifié) ?
2. **Signature personnelle** : **propriétaire strict** (même un admin / superuser ne peut pas récupérer la signature personnelle d'un autre utilisateur via `/media` — conséquence : dans l'admin Django, l'aperçu d'une signature d'un autre utilisateur afficherait une image cassée) — ou **propriétaire + superuser** (exception « bris de glace » pour l'admin) ?
   Recommandation : **propriétaire strict**, cohérent avec « aucune exception ». À confirmer car cela touche l'aperçu admin.
3. **Découpage** : C3-2 en **un seul lot** (les 3 cas ensemble) — recommandé, périmètre limité — ou séparer encore (peu utile ici) ?

## 6. Position dans le séquencement

C3-2 **clôt le chantier média**. Ensuite : **C1** (mise en service `DEBUG=False`, déjà débloquée) puis **E4** (cookies/transport). Étape 2 (ultérieure) : `MEDIA_ROOT` hors webroot + `X-Sendfile`.

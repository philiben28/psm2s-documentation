# L3.4.4 — Solution technique Core / Variantes (proposition d'architecture)

Date : 04/07/2026
Statut : **proposition d'architecture, en attente de validation DT avant
tout développement** (conformément à `IA_RULES` §Avant toute modification :
proposer un plan, ne jamais commencer directement à coder). S'appuie sur
L3.4.1 (cartographie), L3.4.2 (niveaux) et `POLITIQUE-001` (règles
d'évolution), tous validés.

---

## Décision centrale : une instance = un client, jamais de base partagée

Avant les mécanismes de paramétrage eux-mêmes, une décision d'architecture
globale conditionne tout le reste : PSM2S restera un modèle **mono-tenant
par instance** — chaque client dispose de son propre déploiement
(serveur ou sous-domaine dédié, base de données dédiée), exactement comme
aujourd'hui formation et production sont deux instances distinctes du même
code. **Aucune base de données partagée entre plusieurs clients n'est
introduite.**

**Justification** : tu as toi-même indiqué viser quelques établissements,
pas des centaines de clients. Un modèle multi-tenant partagé (une seule
base, un identifiant client sur chaque table, middleware de filtrage
systématique) résoudrait un problème d'échelle que PSM2S n'a pas, au prix
d'une complexité durable (risque d'IDOR inter-clients à chaque nouvelle
table, migrations plus délicates, tests plus lourds). Ce serait
l'inverse du principe déjà appliqué avec succès en Phase 1
(`_get_etab_ids_autorises`, une seule primitive de périmètre) : ici, le
périmètre le plus sûr entre deux clients est bien plus simple à obtenir —
des bases physiquement séparées, pas un filtrage applicatif à maintenir
pour toujours. C'est aussi cohérent avec la Règle d'or de `IA_RULES` (« la
meilleure solution est celle qui répond au besoin avec le minimum de
complexité »).

**Conséquence pratique** : le mécanisme de déploiement d'une nouvelle
instance client est déjà connu — c'est exactement `PROC-001`, appliqué à
un nouveau serveur au lieu d'une mise à jour. Aucune nouvelle procédure de
déploiement n'est nécessaire.

## Niveau 1 — Configuration

**Mécanisme proposé** : étendre le mécanisme déjà en place (`config/env.py`
— `get_env`, `get_env_bool`, `get_env_int`, `get_env_required`) aux
éléments d'identité/branding identifiés en L3.4.1, au lieu des constantes
Python actuelles. Concrètement, dans `config/settings.py` :

- `NOM_ORGANISATION` lu via `get_env('DJANGO_NOM_ORGANISATION', default=...)`
  au lieu d'une constante en dur.
- Domaine (`ALLOWED_HOSTS`, `BASE_URL`) lu depuis l'environnement plutôt
  que codé en dur — chaque instance pose sa propre valeur, exactement
  comme `DJANGO_SECRET_KEY` aujourd'hui.
- Identité d'envoi d'email (`DEFAULT_FROM_EMAIL`, `EMAIL_HOST_USER`) : idem.
- Logo, nom affiché, couleurs de thème : nécessitent d'être accessibles
  aux templates, pas seulement aux settings. Solution : un **context
  processor Django** dédié (ex. `registre.context_processors.instance`)
  exposant ces valeurs dans tous les templates, pour remplacer les
  valeurs actuellement en dur dans `base.html` (`« PSM2S »`, variables
  CSS `--accent`, etc.) par des variables de contexte.

Ce mécanisme est une extension directe de l'existant (même helpers, même
philosophie « secure-by-default » avec valeurs par défaut sûres si une
variable n'est pas posée) — aucun nouveau paradigme technique introduit.

**Déploiement** : ces nouvelles variables suivent exactement `PROC-001`
Étape 3 (Variables d'environnement). Aucune nouvelle procédure requise ;
`DEPLOIEMENT_o2switch.md` sera complété avec la liste de ces variables au
moment de l'implémentation réelle (Phase 4), pas dans cette proposition.

## Niveau 2 — Fonctionnalités activables

**Mécanisme proposé** : une variable d'environnement unique,
`DJANGO_MODULES_ACTIFS` (liste séparée par virgules, ex.
`eau,commissions,duerp,accessibilite,tickets`), lue une fois au chargement
des settings et exposée comme un ensemble Python (`MODULES_ACTIFS`).

Deux gardes nécessaires, pas une seule, pour rester cohérent avec le
principe de défense en profondeur déjà appliqué en Phase 1 (IDOR) :

1. **Garde d'affichage** : le menu de navigation (`base.html`) n'affiche
   un module que s'il figure dans `MODULES_ACTIFS` (via le même context
   processor que le Niveau 1, ou un context processor dédié).
2. **Garde d'accès** : chaque vue du module concerné vérifie que le module
   est actif pour l'instance (petit helper, ex. `module_actif('eau')`
   dans `registre/permissions.py`, aux côtés des primitives de périmètre
   existantes), et redirige/404 sinon. Sans cette deuxième garde, un
   utilisateur connaissant l'URL pourrait accéder à un module
   « désactivé » simplement absent du menu — la même logique qui a motivé
   la primitive `_get_etab_ids_autorises` en Phase 1 s'applique ici.

**Ce qui n'est délibérément pas prévu** : aucune UI d'administration pour
activer/désactiver un module — à ce stade, une variable d'environnement
posée au déploiement suffit très largement (quelques clients, changements
rares). Une interface d'auto-administration serait une sur-conception au
regard du besoin réel actuel.

## Niveau 3 — Personnalisations métier

**Aucun mécanisme construit maintenant**, décision explicite et assumée :
`POLITIQUE-001` prévoit que ce niveau reste rare par construction (étapes
1 à 3 de la séquence de décision épuisées avant d'y arriver). Concevoir un
mécanisme générique de personnalisation métier sans cas réel serait la
sur-conception que tu as toi-même écartée dès L3.4.1. Le jour où un
besoin réel et irréductible se présente, il sera traité au cas par cas, en
suivant `POLITIQUE-001`, avec validation DT systématique — la solution
technique sera conçue à ce moment-là, sur la base d'un besoin concret.

## Ce qui reste à l'autonomie du Lead Developer (Phase 4)

Conformément à `IA_RULES` §Autonomie du Lead Developer, les points suivants
seront tranchés pendant le développement, sans nouvelle validation DT :
nom exact et emplacement du context processor, structure interne du
helper `module_actif`, organisation des fichiers. Seule l'architecture
ci-dessus (mono-tenant par instance, mécanisme env-var + context
processor pour la Configuration, variable + double garde pour les
Modules activables, aucun mécanisme pour le Niveau 3) est soumise à
validation.

## Impact sur l'existant

Aucune régression attendue : les valeurs par défaut des nouvelles
variables reproduisent exactement le comportement actuel (organisation
« Patrimoine ADPEP28 », tous les modules actifs) tant qu'aucune variable
n'est explicitement modifiée. La plateforme actuelle (production,
formation) continue de fonctionner à l'identique sans reconfiguration.

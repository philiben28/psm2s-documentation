## 4. Choix techniques (autonomie Lead Developer)

- **Tickets/Interventions volontairement non câblé dans ce lot.** Ce
  module est référencé depuis DUERP, Accessibilité et Commissions
  (création de ticket liée à une action ou une prescription) ; le
  désactiver proprement suppose aussi de masquer ces liens croisés — un
  point distinct, pour ne pas mélanger deux évolutions dans le même lot
  (IA_RULES §Développement). Le mécanisme (`MODULES_ACTIFS`, y compris
  `'tickets'` dans les modules connus) est déjà prêt à l'accueillir ; seul
  le câblage dans les vues concernées reste à faire, en lot séparé si un
  besoin réel se présente (cf. `POLITIQUE-001`).
- **Garde côté template en plus de la garde côté vue** (affichage des
  onglets conditionnel dans `detail_etablissement.html`) : cohérent avec
  la défense en profondeur déjà appliquée pour l'anti-IDOR en Phase 1 —
  la garde de sécurité réelle reste le décorateur `module_requis` sur la
  vue (Http404 même en accès direct par URL), l'affichage n'est qu'un
  confort UX.
- **`module_requis` bloque tous les utilisateurs sans exception**,
  y compris admin/superuser — différent des décorateurs de rôle
  existants : un module désactivé pour l'instance n'est pas une question
  de permission individuelle, c'est une configuration d'instance.
